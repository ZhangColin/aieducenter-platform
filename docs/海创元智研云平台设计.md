## 一、项目定位

海创元智研云平台是一个 **AI 聚合 SaaS 平台**，面向学校和企业，核心价值：

1. **统一入口**：接入各类大模型（文字对话、生图、生音视频、AI 编程），用户无需分别注册各厂商
2. **智能体**：提供预制和自定义智能体（PPT 生成、文档分析等），处理复杂 AI 任务
3. **费用管理**：虚拟币体系 + 各模型汇率，用户充值后按量使用，企业可控制成员配额
4. **多租户**：支持企业/机构入驻，成员管理，权限配置

参考产品：Manus（任务执行界面、AI 工作流）

### 与 cartisan-boot 的关系

aieducenter-platform 基于 cartisan-boot 框架开发。cartisan-boot 提供 DDD 基建、Web 规范、安全抽象、数据层封装、大模型调用封装等技术基础设施；aieducenter-platform 在此之上实现所有业务逻辑。

两个项目并行开发，开发期通过 Gradle Composite Build 联调。

## 二、技术栈

### 后端

依赖 cartisan-boot 提供的技术基座，业务项目额外引入：

| 层面 | 选型 | 用途 |
|------|------|------|
| 框架基座 | cartisan-boot | DDD、Web、Security、Data、AI、Event、Test |
| 消息队列 | Redis Streams（初期）→ Kafka（规模增长后） | 计费异步处理、事件分发 |
| 定时任务 | Spring Scheduler（初期）→ XXL-Job（规模增长后） | 对账、配额重置、数据统计 |
| 对象存储 | cartisan-storage（阿里云 OSS / MinIO） | 用户上传文件、AI 生成的图片/音视频 |
| 支付 | cartisan-payment（微信支付 / 支付宝） | 用户充值 |
| 缓存 | Redis | 余额缓存、限流、会话上下文缓存 |

### 前端

| 层面 | 选型 | 理由 |
|------|------|------|
| 框架 | Next.js 15（App Router） | SSR 支持落地推广页面，Vercel AI SDK 原生支持流式对话 |
| 语言 | TypeScript | 类型安全，与后端 OpenAPI 生成的类型对齐 |
| UI 组件 | shadcn/ui + Tailwind CSS | 美观、源码可控、后台和前台统一一套、AI 生成代码质量高 |
| 状态管理 | Zustand | 轻量，适合 AI 对话等场景的客户端状态 |
| API 通信 | OpenAPI 自动生成的 TypeScript 客户端 | 前后端类型安全对齐，后端改接口前端编译期发现 |
| Monorepo | pnpm workspace | 轻量，共享 UI 组件和 API 客户端 |

## 四、前端架构

### 4.1 两个前端应用

| 应用 | 路径 | 面向 | 部署 |
|------|------|------|------|
| **web** | `/web` | 终端用户 + 企业管理员 | user.aieducenter.com |
| **admin** | `/admin` | 平台运营团队 | ops.aieducenter.com |

**web 应用**中，用户和企业管理员共用同一个应用。普通用户看到 AI 对话、智能体、个人中心等功能；企业管理员通过头像菜单进入管理视图（`/manage/*`），看到成员管理、用量统计等功能。通过角色控制菜单可见性和路由权限。

### 4.2 共享层设计

**packages/ui** — 基于 shadcn/ui 的共享组件库：
- 从 shadcn/ui 复制基础组件（Button、Input、Dialog、Table、Form 等）
- 在此基础上封装业务通用组件（如 ModelSelector、CoinBalance、UsageChart）
- web 和 admin 都从这里引用，保证视觉风格统一

**packages/api-client** — 自动生成的 API 客户端：
- 后端 SpringDoc 输出 OpenAPI 3.1 规范
- 使用工具（如 openapi-typescript）自动生成 TypeScript 类型和请求函数
- 前端调用 API 有完整类型提示，后端改接口前端编译期报错

### 4.3 AI 对话的前端关键技术

- **流式渲染**：Vercel AI SDK 的 `useChat` hook，原生支持 SSE 流式输出
- **Markdown 渲染**：支持代码高亮、LaTeX 公式、表格
- **多模态展示**：图片/音频/视频/代码块内联展示
- **模型切换**：对话顶部下拉选择模型

### 4.4 后续扩展

将来如需支持小程序等端，后端 API 不变，前端单独加一个端（使用 Taro 等框架），不影响现有结构。

## 六、多端设计与用户权限体系

### 6.1 用户层级

```
Platform（平台）
  │
  ├── Platform Admin Users（平台运营人员）
  │     角色：超级管理员 / 运营 / 客服 / 财务
  │     入口：admin 应用（ops.aieducenter.com）
  │
  └── Tenant（租户/企业/机构）
        │
        ├── Tenant Admin Users（企业管理员）
        │     角色：企业管理员 / 部门管理员
        │     入口：web 应用的管理视图（/manage/*）
        │
        └── Tenant Member Users（企业成员/终端用户）
              角色：普通用户 / VIP 用户
              入口：web 应用（user.aieducenter.com）
```

### 6.2 关键设计

- 每个用户属于一个租户（个人用户 = 单人租户，自动创建）
- 租户是计费主体，虚拟币余额挂在租户下
- 企业可以给成员设置每日/每月用量配额
- 平台管理员可跨租户操作
- web 应用内，企业管理员通过头像菜单 → "管理后台" 进入管理视图，无需跳转到另一个站点

## 七、核心业务模型

以下是主要业务的模型设计概览。每个模型的详细设计在实现对应 Epic 时，通过 SOP 的 Phase 1-3 产出完整的 spec 文档。

### 7.1 虚拟币计费模型

#### 核心概念

```
用户充值人民币 → 获得虚拟币（"智慧币"）
                    │
                    ▼
使用 AI 能力 → 消耗虚拟币
                    │
           ┌────────┼────────┐
           ▼        ▼        ▼
        GPT-4o   Claude   通义千问
        汇率 A   汇率 B    汇率 C
```

#### 计费公式

```
本次消耗虚拟币 = (input_tokens × 输入汇率 + output_tokens × 输出汇率)

其中：
  输入汇率 = 该模型每千 input tokens 对应的虚拟币数
  输出汇率 = 该模型每千 output tokens 对应的虚拟币数
```

示例：

| 模型 | 输入汇率（虚拟币/千token） | 输出汇率（虚拟币/千token） |
|------|-------------------------|-------------------------|
| GPT-4o | 0.5 | 1.5 |
| Claude Sonnet 4 | 0.6 | 1.8 |
| 通义千问-Max | 0.2 | 0.4 |
| DALL-E 3 (生图) | — | 40/张 |
| Suno (生音乐) | — | 20/首 |

充值比例：1 元 = 100 虚拟币（平台可调整）

#### 计费流程

```
用户发起 AI 请求
    │
    ▼
[1] 查询用户所属租户 + 用户配额
    ▼
[2] 查询目标模型的当前汇率
    ▼
[3] 预估最大消耗 = max_tokens × output_rate
    ▼
[4] Redis 原子操作：检查余额 + 检查配额 + 预扣
    ├─ 余额不足 → 返回"余额不足，请充值"
    ├─ 配额超限 → 返回"已达今日/本月使用上限"
    └─ 预扣成功 → 继续
        ▼
[5] 调用大模型（SSE 流式），同时累计实际 token 消耗
    ▼
[6] 流结束 → 计算实际消耗虚拟币
    ▼
[7] Redis 原子操作：实际扣减（多退少补）
    ▼
[8] 发布 CoinConsumedEvent 领域事件
    ▼
[9] 异步：写入消耗流水 + 更新用户配额
    ▼
[10] 定时对账：Redis 余额 ↔ DB 流水汇总
```

#### 汇率管理策略

- 汇率可预设未来生效时间
- 消耗时快照汇率 ID，事后核对有据可查
- 平台利润 = 上游模型成本 + 加价
- 后续可扩展租户差异化定价

### 7.2 AI 对话流程

```
用户选择模型 → 输入消息
    │
    ▼
前端调用 POST /api/chat（携带 conversationId、model、messages）
    │
    ▼
AI Gateway Context:
  [1] 鉴权 + 租户识别
  [2] 模型路由（如果用户未指定具体模型，按策略选择）
  [3] → Billing 预扣
  [4] → cartisan-ai ModelProvider.chatStream()
  [5] SSE 流式输出给前端
  [6] 流结束 → 回调 TokenUsage
  [7] → Billing 实际扣减
  [8] → Conversation 保存消息
    │
    ▼
前端逐字渲染（Vercel AI SDK useChat）
```

### 7.3 会话管理模型

```
Conversation（会话/聚合根）
  - id
  - userId
  - tenantId
  - title（自动从第一条消息摘要生成）
  - model（当前使用的模型）
  - createdAt
  - updatedAt

Message（消息/实体，属于 Conversation 聚合）
  - id
  - conversationId
  - role（USER / ASSISTANT / SYSTEM）
  - content
  - model（生成此消息的模型）
  - tokenUsage（input/output tokens）
  - createdAt
```

初期实现简单：前端传完整消息历史，后端透传给模型。后续根据需要加上下文窗口管理（截断/摘要）。

| **Epic 8: 文件与多模态** | 对话中上传文件、图片。消息中展示图片/文件。集成 cartisan-storage | M |

| **Epic 10: 智能体** | Agent Context：智能体定义与配置、工具注册与调用、多步任务编排。web 端智能体市场与执行界面 | XL |
| **Epic 11: 落地推广页** | web 端公开页面：产品介绍、功能展示、定价页面（SEO 优化） | S |

### 8.3 说明

- MVP（Epic 1-4）完成后，系统可用：用户登录 → 选模型 → AI 对话 → 查看历史
- MVP 阶段暂不计费，可设置为免费使用或固定 Token 额度
- 增长阶段的 Epic 按业务需要选择实施顺序，不强制先后
- 每个 Epic 实施时，通过 SOP 的 Phase 0 进一步拆解为 Feature，再通过 Phase 1-4 产出详细 spec 和代码
